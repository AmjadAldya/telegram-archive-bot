# AI Project Context

## 1. Executive Overview
- Project name: telegram-archive-bot. The README also describes it as Telegram Archive Bot v2.
- Confirmed summary: this is a small Python Telegram bot built with Pyrogram that currently responds to bot commands and runs a limited archive routine over the owner chat history.
- Main business purpose: automate Telegram media archiving so a user or operator can capture message content without manually browsing and copying it.
- Core value proposition: reduce manual effort for archiving Telegram media and create a base for future resume, queue, upload, and admin features.
- Current apparent project status: early-stage MVP / scaffold. One archive path is implemented, but several surrounding modules are placeholders.
- High-level system summary: one bot process starts a Pyrogram client, registers message handlers, and routes the /archive command to an async archive engine. Database, queue, resume, downloader, and uploader modules exist but are not yet connected to the runtime flow.

## 2. Product and Business Context
- Problem solved: manual Telegram archiving is tedious, fragile, and easy to lose track of when large message histories are involved.
- Target users or audience: likely a single operator, bot administrator, or internal user who wants to preserve Telegram media from a chat or saved messages account. Confirmed admin logic exists only as configuration, not enforcement.
- Main use cases: start the bot, verify it is alive, trigger an archive run, and eventually extend that run with resume, queue, and upload/download behavior.
- Key business workflows: bot startup, command handling, archive processing, and eventual persistence of archive state.
- Important product terminology: archive bot, archive engine, queue system, resume state, admin ID, media download, media upload.
- Known business rules found in code and docs:
  - Confirmed: /start replies with Archive Bot running.
  - Confirmed: /archive replies with Starting archive... and then invokes the archive engine.
  - Confirmed: the archive engine scans chat history for me using Pyrogram and counts media messages.
  - Confirmed: the current loop stops after more than 20 media items are encountered.
  - Confirmed: ADMIN_ID is loaded from the environment, but no authorization gate uses it yet.
  - Confirmed: DATABASE_URL defaults to sqlite:///data.db.
- Non-technical goals inferred from the repository: automation, low operational overhead, Dockerized deployment, and a path toward production readiness.

## 6. Architecture
- Architectural style: event-driven Telegram bot with a thin handler layer and modular placeholders for future archive pipeline concerns.
- Main layers and modules:
  - Entrypoint: [app/main.py](app/main.py).
  - Telegram client lifecycle: [app/bot/client.py](app/bot/client.py).
  - Command handlers: [app/bot/handlers.py](app/bot/handlers.py).
  - Archive logic: [app/archive/engine.py](app/archive/engine.py).
  - Configuration: [app/config/settings.py](app/config/settings.py).
  - Services: [app/services/logger.py](app/services/logger.py) and [app/services/queue.py](app/services/queue.py).
  - Persistence scaffolding: [app/database/db.py](app/database/db.py) and [app/database/models.py](app/database/models.py).
- How data flows through the system:
  - The process starts in [app/main.py](app/main.py).
  - [app/bot/client.py](app/bot/client.py) creates a singleton Pyrogram Client and starts it.
  - [app/bot/handlers.py](app/bot/handlers.py) registers /start and /archive handlers on that client.
  - The /archive handler replies to the user and calls [app/archive/engine.py](app/archive/engine.py).
  - The archive engine iterates over client.get_chat_history('me') and counts media messages until the loop is stopped after 20 media items.
- Frontend-backend interaction: not applicable in the usual web sense. The Telegram chat itself is the user interface.
- Service boundaries: the bot client owns Telegram connectivity, while the archive engine owns the current processing logic. Queue, database, downloader, resume, and uploader are separate conceptual boundaries but are not yet integrated.
- Event flow and job flow:
  - Bot command event arrives from Telegram.
  - Pyrogram dispatches to the registered handler.
  - Handler replies immediately and starts archive processing.
  - Archive loop logs start and completion and counts media.
- Important design patterns used:
  - Async coroutines throughout the runtime path.
  - Module-level singleton client and logger objects.
  - Separation of concerns by folder, even where modules are still placeholders.
- Coupling observations:
  - handlers.py imports the archive engine directly, so the command path is tightly coupled to that module.
  - db.py creates an engine on import, but nothing currently consumes it.
  - queue.py exposes a global asyncio.Queue, but no runtime code uses it yet.
- Plain-English architecture: this is currently a small Telegram bot that starts, listens for commands, and runs a simple archive scan. The repo is already structured like a fuller archive system, but most supporting subsystems are planned rather than active.

- /start command flow:

  - There are no additional checks, permissions, or state changes.
- /archive command flow:
  - Telegram dispatches /archive to the handler in [app/bot/handlers.py](app/bot/handlers.py).
  - The handler replies with Starting archive....

  - The engine logs Archive started.
  - The engine iterates over client.get_chat_history('me').
  - Each message with media increments a counter.
  - When the counter becomes greater than 20, the loop breaks.
  - The engine logs the number of processed items.

## 23. Glossary
- Archive bot: the Telegram bot service that starts an archive process on command.
- Archive engine: the active processing logic in [app/archive/engine.py](app/archive/engine.py) that scans chat history and counts media.
- Resume state: planned persisted state that would allow an interrupted archive job to continue later.
- Queue system: the in-memory async queue in [app/services/queue.py](app/services/queue.py) that is intended for task orchestration.
- Admin ID: an integer environment value intended to identify an operator, but not yet enforced.
- Me chat: the Telegram self chat used by Pyrogram when calling get_chat_history('me').
- Placeholder module: a file that documents intended future functionality but does not yet implement it.
- Pyrogram Client: the Telegram client object created in [app/bot/client.py](app/bot/client.py).
